   ```typescript
   // DTOs
   import { UserResDto } from './dto/user-res.dto';

   // Services
   import { AuthService } from './auth.service';

   // Guards/Decorators
   import { JwtAuthGuard } from '@/common/guards';

   // Interfaces/Types

   ```
   - Import grupları boş satırlarla ayrılmış
   - Her grup içinde alfabetik sıralama

## Dev Notes

### Architecture Patterns and Constraints

**Base Class Design Philosophy:**
- Base classes capture common patterns across the codebase
- Follow DRY principle: define once, extend everywhere
- Generic type support for type-safety: `BaseResponseDto<T>`
- All base classes properly decorated for validation and Swagger
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Code-Organization]

**Import Organization Standard:**
```typescript
// 1. Libraries (External packages - @nestjs, class-validator, etc.)
import { Controller, Get, Post } from '@nestjs/common';
import { IsString, IsOptional } from 'class-validator';

// 2. DTOs (Data Transfer Objects)
import { CreateUserDto } from './dto/create-user.dto';
import { UserResDto } from './dto/user-res.dto';

// 3. Services (Business logic services)
import { UsersService } from './users.service';
import { AuthService } from '../auth/auth.service';

// 4. Guards/Decorators (Custom decorators, guards, interceptors)
import { JwtAuthGuard } from '@/common/guards';
import { Public } from '@/common/decorators';

// 5. Interfaces/Types (Type definitions, interfaces)
import type { JwtPayload } from './interfaces/jwt-payload.interface';
import type { UserRole } from './types/user-role.type';
```
- Each group separated by blank line
- Within group: alphabetically sorted
- Type-only imports use `import type` syntax
- [Source: Epic 8.5 Requirements]

**Guard Placement Strategy:**
- Controller-level guards apply to all endpoints in that controller
- Use `@Public()` decorator to exclude specific endpoints from authentication
- Method-level guards only for endpoint-specific overrides
- Standard pattern: `@UseGuards(JwtAuthGuard, PermissionsGuard)` at class level
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Controller-Pattern]

**Entity Inheritance Pattern:**
```typescript
// BaseEntity: Minimal common fields
export abstract class BaseEntity {
  id: string;
  createdAt: Date;
  updatedAt: Date;
  deletedAt?: Date; // Soft delete support
}

// BaseAuditEntity: Extends BaseEntity with audit fields
export abstract class BaseAuditEntity extends BaseEntity {
  createdBy?: string;
  updatedBy?: string;
}

// Application Entity: Extends appropriate base
export class User extends BaseAuditEntity {
  email: string;
  // ... user-specific fields
}
```
- Use BaseEntity for simple entities (Role, Permission)
- Use BaseAuditEntity for entities needing audit trail (User, File)
- No Prisma schema migration needed (fields already exist, just refactoring)
- [Source: Epic 8.5 Requirements]

**DTO Base Class Patterns:**
```typescript
// Query DTOs extend BaseQueryDto
export class GetUsersQueryDto extends BaseQueryDto {
  @IsOptional()
  @IsEnum(UserStatus)
  status?: UserStatus;
}

// Response DTOs align with BaseResponseDto structure
// Actual wrapping done by interceptor, DTOs return raw data
export class UserResDto {
  id: string;
  email: string;
  // ... fields with @Expose(), @ApiProperty()
}
```
- BaseQueryDto provides pagination, sorting, filtering
- Response DTOs don't need to extend base (interceptor wraps)
- But structure should align with BaseResponseDto for consistency
- [Source: docs/architecture/api-contracts.md#Response-Format-Standards]
