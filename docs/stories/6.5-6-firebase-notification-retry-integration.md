# Story 6.5.6: Firebase Notification Retry Integration

Status: review

## Story

As a developer,
I want FirebaseService'te retry mekanizmasını Common RetryService ile entegre etmek,
so that push notification gönderimlerinde geçici network hataları ve Firebase API hatalarında otomatik retry yapılsın ve kritik bildirimlerin güvenilirliği artsın.

## Acceptance Criteria

1. **AC-6.5.6.1:** FirebaseService Dependencies Update
   - `src/modules/notifications/services/firebase.service.ts` güncellendi
   - Constructor'da RetryService inject edilmiş
   - RetryService (Story 6.5-2'den) import edilmiş: `src/common/services/retry.service.ts`
   - ConfigService inject edilmiş (retry configuration için)

2. **AC-6.5.6.2:** FirebaseService Send Retry Integration
   - sendPush() method'u RetryService ile wrap edilmiş
   - Firebase Admin SDK send() çağrısı retry ile sarmalanmış
   - RetryService.executeWithRetry() kullanılır:
     ```typescript
     const response = await this.retryService.executeWithRetry(
       async () => await admin.messaging().send(message),
       {
         maxAttempts: this.firebaseRetryMaxAttempts, // 5 (configurable)
         baseDelay: this.firebaseRetryBaseDelay, // 2000ms (configurable)
         context: `Firebase Notification: ${deviceToken.substring(0, 20)}...`
       }
     );
     ```
   - Max attempts: 5 (critical communication)
   - Base delay: 2000ms (external service)

3. **AC-6.5.6.3:** Error Handling - Firebase Specific Errors
   - Network errors: Retry (RetryService handles)
   - Firebase server errors (5xx, 'messaging/unavailable'): Retry
   - Invalid token ('messaging/invalid-registration-token'): Don't retry, throw immediately
   - Invalid message format (400): Don't retry, throw immediately
   - Quota exceeded (429): Retry (RetryService handles with exponential backoff)
   - Retry exhausted: RetryService throws last error, preserve existing error handling

4. **AC-6.5.6.4:** Error Classification Logic
   - Before retry, check Firebase error codes:
     - `messaging/invalid-registration-token` → throw immediately (skip retry)
     - `messaging/invalid-argument` → throw immediately (skip retry)
     - `messaging/unavailable` → let RetryService handle (retry)
     - `messaging/server-error` → let RetryService handle (retry)
     - Network errors (ETIMEDOUT, ECONNRESET) → let RetryService handle (retry)
   - Preserve existing error messages and Sentry logging

5. **AC-6.5.6.5:** Environment Configuration
   - `.env.example` dosyasına yeni environment variables eklendi:
     ```
     # Firebase Service Retry Configuration
     FIREBASE_RETRY_MAX_ATTEMPTS=5     # Max retry attempts (critical communication)
     FIREBASE_RETRY_BASE_DELAY=2000    # Base delay (ms)
     ```
   - ConfigService integration:
     - `FIREBASE_RETRY_MAX_ATTEMPTS`: Number, default 5
     - `FIREBASE_RETRY_BASE_DELAY`: Number, default 2000

6. **AC-6.5.6.6:** Logging Integration
   - Send start: `[FirebaseService] Sending push notification to device: ${deviceToken.substring(0, 20)}...` (existing log)
   - Retry attempts: `[RetryService] Retry attempt 2/5 for Firebase Notification: ${deviceToken.substring(0, 20)}...` (RetryService logging)
   - Success: `[FirebaseService] Push notification sent successfully (messageId: ${response})` (existing log)
   - Invalid token: `[FirebaseService] Invalid device token: ${deviceToken.substring(0, 20)}...` (existing warning)
   - Final failure: `[FirebaseService] Failed to send push notification: ${error.message}` (existing error log)
   - Preserve existing debug and error logs
   - Preserve existing Sentry error tracking

7. **AC-6.5.6.7:** Unit Tests - FirebaseService
   - Test file: `src/modules/notifications/services/firebase.service.spec.ts` güncellendi
   - Test scenarios:
     - Push notification send successful on first attempt
     - Push notification retry on network error ('messaging/unavailable')
     - Push notification retry on server error
     - Push notification no retry for invalid token ('messaging/invalid-registration-token')
     - Push notification retry exhaustion (throw exception after max attempts)
     - Sentry logging preserved during retry
   - Mock RetryService, Firebase Admin SDK, ConfigService

8. **AC-6.5.6.8:** Configuration Service Integration
   - ConfigService injected in constructor
   - Read retry configuration from environment:
     - `this.firebaseRetryMaxAttempts = this.configService.get<number>('FIREBASE_RETRY_MAX_ATTEMPTS', 5)`
     - `this.firebaseRetryBaseDelay = this.configService.get<number>('FIREBASE_RETRY_BASE_DELAY', 2000)`

## Dev Notes

### Architecture Patterns and Constraints

**Service Integration Pattern:**
- FirebaseService, RetryService'i dependency injection ile alır
- RetryService: Exponential backoff retry (Story 6.5-2'den)
- ConfigService: Retry configuration management
- [Source: docs/tech-spec-epic-6.5.md#Integration-Points]

**Retry Pattern:**
- Wrap Firebase Admin SDK send() call with retry
- Exponential backoff: 2^(attempt-1) * baseDelay
- Max attempts: 5 for push notifications (configurable)
- Base delay: 2000ms (critical communication, same as email/SMS)
- Context-based logging for debugging
- [Source: docs/tech-spec-epic-6.5.md#Retry-Workflow]

**Error Handling Pattern:**
- Firebase validation errors (invalid-registration-token, invalid-argument): Don't retry, throw immediately
- Network errors (messaging/unavailable): Retry via RetryService
- Server errors (5xx): Retry via RetryService
- Quota exceeded (429): Retry via RetryService (exponential backoff helps)
- Retry exhausted: Throw last error, preserve existing error handling and Sentry tracking
- [Source: docs/tech-spec-epic-6.5.md#Error-Handling]

**Configuration Management:**
- Environment variables: FIREBASE_RETRY_MAX_ATTEMPTS (default 5), FIREBASE_RETRY_BASE_DELAY (default 2000ms)
- ConfigService ile configuration management
- Same retry attempts as email/SMS (5 attempts) for critical communication
- [Source: docs/tech-spec-epic-6.5.md#Configuration-Management]

**Module Structure:**
- FirebaseService: `src/modules/notifications/services/firebase.service.ts`
- RetryService: `src/common/services/retry.service.ts` (Story 6.5-2)
- [Source: docs/architecture/project-structure.md#modules]

**Import Organization:**
- 8-group import order: Libraries → DTOs → Services → Repositories → Entities → Interfaces → Enums → Events
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Import-Organization]

### Learnings from Previous Story

**From Story 6.5-5-mail-sms-services-retry-integration (Status: done)**

- **RetryService Usage Pattern:**
  - RetryService.executeWithRetry() ile operation wrap et
  - Retry options: maxAttempts, baseDelay, context
  - Context string for logging: `Firebase Notification: ${deviceToken.substring(0, 20)}...`
  - Use `RetryService` from `@/common/services/retry.service`
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Dev-Agent-Record]

- **New Service Created - RetryService:**
  - RetryService already implemented in Story 6.5-2
  - Available at: `src/common/services/retry.service.ts`
  - Exported from CommonModule
  - Usage: Inject via constructor: `private readonly retryService: RetryService`
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Learnings-from-Previous-Stories]

- **Exponential Backoff:**
  - Formula: `2^(attempt-1) * baseDelay`
  - Delay sequence: 0ms → 2000ms → 4000ms → 8000ms... (for baseDelay=2000ms)
  - First attempt: Immediate (no delay)
  - Push notification operations need longer base delay (2000ms, same as email/SMS)
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Architecture-Patterns]

- **Retry Logging:**
  - Each attempt: `[context] Attempt 1/5`
  - Retry delay: `[context] Waiting 2000ms before retry attempt 2`
  - Success on retry: `[context] Succeeded on attempt 2/5`
  - Final failure: `[context] All 5 attempts failed`
  - Context string critical for debugging
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Logging-Integration]

- **Configuration Pattern:**
  - Environment variables: FIREBASE_RETRY_MAX_ATTEMPTS, FIREBASE_RETRY_BASE_DELAY
  - ConfigService integration with defaults
  - Higher retry attempts for critical communication (5 vs 3)
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Environment-Configuration]

- **Error Handling:**
  - RetryService throws last error after max attempts
  - Wrap RetryService error with custom exception or preserve existing error handling
  - Non-retryable errors (4xx validation): Throw immediately, skip retry
  - Preserve existing error logging and Sentry tracking
  - [Source: stories/6.5-5-mail-sms-services-retry-integration.md#Error-Handling-Pattern]

- **Code Quality Improvement:**
  - Single responsibility principle (RetryService handles retry)
  - Cleaner code, easier to maintain
  - Consistent retry behavior across services
  - Preserve existing error handling patterns (Sentry integration)

- **Pending Review Items:** None affecting this story

**Key Takeaways:**
- Story 6.5-6 integrates RetryService into FirebaseService
- Higher retry attempts (5) for critical communication (same as email/SMS)
- Longer base delay (2000ms) for external service operations
- Error classification: Firebase validation errors skip retry, network and server errors retry
- Preserve existing error handling patterns (Sentry error tracking, error messages)
- ConfigService integration for retry configuration
