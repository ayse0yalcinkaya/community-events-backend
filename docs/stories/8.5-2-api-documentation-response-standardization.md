# Story 8.5-2: API Documentation & Response Standardization

Status: done

## Story

As a developer,
I want enhanced ApiEndpoint decorator with ApiBody integration and a global response interceptor,
So that API documentation is consistent and responses are in standard format.

## Acceptance Criteria

1. **AC-8.5-2.1:** ApiEndpoint decorator geliştirilmiş (`src/common/decorators/api-endpoint.decorator.ts`)
   - ApiBody entegrasyonu eklendi
   - Şu parametreleri destekliyor:
     ```typescript
     interface ApiEndpointOptions {
       summary: string;
       type?: Type<any>;
       status?: number;
       consumes?: string; // 'application/json' | 'multipart/form-data'
       bodySchema?: SchemaObject; // ApiBody için schema
       isArray?: boolean;
       isPaginated?: boolean;
       isPublic?: boolean;
     }
     ```
   - File upload için özel destek:
     ```typescript
     @ApiEndpoint('Dosya yükle', {
       type: FileResDto,
       consumes: 'multipart/form-data',
       status: 201,
       bodySchema: {
         type: 'object',
         properties: {
           files: {
             type: 'array',
             items: { type: 'string', format: 'binary' }
           }
         }
       }
     })
     ```

2. **AC-8.5-2.2:** ApiCrud decorator geliştirilmiş (`src/common/decorators/api-crud.decorator.ts`)
   - CRUD operasyonları için body schema desteği eklendi
   - File upload senaryolarını destekliyor
   - Standard CRUD decorator'lar bodySchema parametresi kabul ediyor

3. **AC-8.5-2.3:** Tüm controller'larda ApiBody kaldırılmış
   - ApiEndpoint veya ApiCrud kullanılıyor
   - Özellikle file upload endpoint'lerinde ApiBody yerine ApiEndpoint bodySchema parametresi kullanılıyor
   - Controller'larda manual ApiBody decorator'ı kalmamış

4. **AC-8.5-2.4:** Global Response Interceptor oluşturulmuş
   - `src/common/interceptors/transform-response.interceptor.ts`
   - Tüm success response'ları şu formata çeviriyor:
     ```typescript
     {
       success: true,
       status: 200,
       data: <actual-data>,
       message: string
     }
     ```
   - Paginated response'lar için:
     ```typescript
     {
       success: true,
       status: 200,
       data: <array>,
       count: number,
       message: string
     }
     ```
   - Error response'lar için:
     ```typescript
     {
       success: false,
       status: 4xx/5xx,
       message: string,
       errors?: array
     }
     ```

5. **AC-8.5-2.5:** Global Interceptor app'e register edilmiş
   - `main.ts` veya `app.module.ts` içinde
   - `app.useGlobalInterceptors(new TransformResponseInterceptor())`
   - Tüm endpoint'lere otomatik uygulanıyor

6. **AC-8.5-2.6:** Mevcut controller'lar güncellendi
   - Manual response wrapping kaldırıldı
   - Service'ler direkt data dönüyor (wrapping olmadan)
   - Interceptor otomatik wrapping yapıyor
   - Success message'lar optional olarak service'den veya interceptor'dan geliyor

7. **AC-8.5-2.7:** Response Interceptor i18n ile entegre
   - Constructor'da `I18nService` inject edilmiş
   - Default message'lar i18n ile translate ediliyor
   - Service'den gelen custom message'lar preserve ediliyor

8. **AC-8.5-2.8:** Interceptor error handling ile uyumlu
   - Exception filters ile çakışma yok
   - Error response'lar exception filter'dan geliyor
   - Interceptor sadece success response'ları wrap ediyor

9. **AC-8.5-2.9:** Response type detection otomatik
   - Array response detection: `Array.isArray(data)`
   - Paginated response detection: `data.items && data.count` pattern
   - Single object response: Default format
   - Swagger documentation otomatik güncelleniyor

10. **AC-8.5-2.10:** Build ve type checking başarılı
    - `npm run build` hata vermeden çalışıyor
    - `npm run type-check` başarılı
    - Hiçbir TypeScript error yok
    - Swagger UI çalışıyor ve response formatları doğru görünüyor

## Dev Notes

### Architecture Patterns and Constraints

**Response Interceptor Pattern:**
- NestJS interceptors use RxJS operators to transform responses
- Interceptors run AFTER controller method execution but BEFORE response is sent
- Use `map()` operator from `rxjs/operators` to transform observable
- Access execution context for metadata (status code, route info)
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Interceptor-Pattern]

**Standard Response Format:**
```typescript
// Success Response
{
  success: true,
  status: 200 | 201 | 204,
  data: T | T[],
  message: string,
  count?: number  // Only for paginated responses
}

// Error Response (from exception filter)
{
  success: false,
  status: 4xx | 5xx,
  message: string,
  errors?: string[]  // Validation errors
}
```
- This format matches `createApiResponseClass()` and `createPaginatedApiResponseClass()` from Epic 8
- [Source: docs/tech-spec-epic-8.md#Data-Models]

**Paginated Response Detection:**
```typescript
// Service returns this structure for pagination
{
  items: T[],
  count: number
}

// Interceptor transforms to:
{
  success: true,
  status: 200,
  data: T[],
  count: number,
  message: string
}
```
- Services should return `{ items, count }` pattern for paginated data
- Interceptor detects this pattern and transforms accordingly
- [Source: Epic 8.5 Requirements]

**ApiBody Integration Strategy:**
- ApiBody decorator needed for Swagger to document request body
- For file uploads: `@ApiConsumes('multipart/form-data')` + ApiBody with binary schema
- ApiEndpoint should compose these decorators internally
- Removes boilerplate from controller methods
- [Source: docs/epics/epic-8.5-technical-refinement-code-quality.md#Story-8.5-2]

**i18n Integration:**
- Interceptor should translate default success messages
- Service can provide custom message which interceptor preserves
- Message keys: `success.OPERATION_SUCCESSFUL`, `success.CREATED`, `success.UPDATED`, `success.DELETED`
- Accept-Language header determines translation
- [Source: Epic 7.1 i18n implementation]

### Project Structure Notes

**Interceptor Location:**
- Following NestJS convention: `src/common/interceptors/`
- Aligns with existing interceptor pattern (request-response logging)
- Global registration in `main.ts` consistent with other global features
- [Source: Project structure analysis]

**Response Transformation Flow:**
```
Controller Method
    ↓
Service Returns Raw Data
    ↓
Controller Returns Service Data
    ↓
TransformResponseInterceptor (intercept)
    ↓
Detect Response Type (single, array, paginated)
    ↓
Wrap with Standard Format
    ↓
Translate Message (i18n)
    ↓
Return Transformed Response
    ↓
Client Receives Consistent Format
```

**Error Handling Flow:**
```
Controller Method Throws Exception
    ↓
Exception Filter Catches
    ↓
Exception Filter Returns Error Format
    ↓
Interceptor SKIPS (errors bypass interceptor)
    ↓
Client Receives Error Response
```
- Interceptors do NOT run on exceptions
- Exception filters handle error formatting
- No conflict between interceptor and exception filter
- [Source: NestJS documentation, execution order]

**ApiBody Integration Pattern:**
```typescript
// Before (manual ApiBody)
@ApiBody({ schema: { type: 'object', properties: { ... } } })
@ApiEndpoint('Upload file', { type: FileResDto })
@Post('upload')
async upload() {}

// After (integrated into ApiEndpoint)
@ApiEndpoint('Upload file', {
  type: FileResDto,
  consumes: 'multipart/form-data',
  bodySchema: {
    type: 'object',
    properties: {
      file: { type: 'string', format: 'binary' }
    }
  }
})
@Post('upload')
async upload() {}
```
- Single decorator handles all Swagger documentation
- Cleaner controller code
- Consistent pattern across all endpoints
