# Story 6.5.2: Common Retry Service (Exponential Backoff)

Status: ready-for-dev

## Story

As a developer,
I want proje genelinde kullanılabilir retry mechanism,
so that tüm async operasyonlar (S3 upload, external API calls, database operations) için transient failures otomatik retry edilsin.

## Acceptance Criteria

1. **AC-6.5.2.1:** RetryService oluşturulmuş
   - `src/common/services/retry.service.ts` dosyası oluşturulmuş
   - Configurable max attempts (default: 3)
   - Configurable base delay (default: 1000ms)
   - Exponential backoff: 2^(attempt-1) * baseDelay
   - Generic type support: `<T>` for any return type

2. **AC-6.5.2.2:** executeWithRetry() Method implement edilmiş
   - Method signature: `executeWithRetry<T>(operation: () => Promise<T>, options?: RetryOptions): Promise<T>`
   - RetryOptions interface:
     - `maxAttempts?: number` (default: 3)
     - `baseDelay?: number` (default: 1000ms)
     - `context?: string` (for logging: 'S3 upload: invoice-001.pdf')
     - `onRetry?: (attempt: number, error: Error) => void` (optional callback)
   - Retry logic:
     - Attempt 1: Execute immediately
     - Attempt 2: Wait baseDelay (2^0 * baseDelay), retry
     - Attempt 3: Wait 2*baseDelay (2^1 * baseDelay), retry
     - Success → Return result
     - All attempts fail → Throw last error

3. **AC-6.5.2.3:** Exponential Backoff Calculation implement edilmiş
   - Formula: `Math.pow(2, attempt - 1) * BASE_DELAY`
   - Delay sequence: 0ms → baseDelay → 2*baseDelay → 4*baseDelay...
   - First attempt: No delay (immediate execution)
   - Subsequent attempts: Exponential delay before retry

4. **AC-6.5.2.4:** Logging Integration implement edilmiş
   - Use Common LoggerService (Epic 7.3)
   - Log each attempt: `[context] Attempt 1/3`
   - Log retry delay: `[context] Waiting 1000ms before retry attempt 2`
   - Log success on retry: `[context] Succeeded on attempt 2/3`
   - Log final failure: `[context] All 3 attempts failed`
   - Context string for debugging and logging

5. **AC-6.5.2.5:** Optional Retry Callback implement edilmiş
   - `onRetry?: (attempt: number, error: Error) => void` callback support
   - Callback invoked before each retry attempt
   - Provides attempt number and error for custom handling
   - Optional: Service works without callback

6. **AC-6.5.2.6:** CommonModule Export implement edilmiş
   - RetryService exported from `src/common/services/index.ts`
   - Available to all modules via CommonModule import
   - CommonModule üzerinden tüm modüllere export edilir

7. **AC-6.5.2.7:** Unit Tests implement edilmiş
   - Test file: `src/common/services/__tests__/retry.service.spec.ts`
   - Test coverage: > 80%
   - Test scenarios:
     - Retry execution with success scenarios
     - Retry execution with failure scenarios
     - Exponential backoff delay calculation
     - Configurable max attempts and base delay
     - Logging calls verification
     - Callback invocation
     - Generic type support (string, number, object, array)
     - Final failure throws last error

8. **AC-6.5.2.8:** Integration Tests implement edilmiş
   - RetryService with mock failures
   - Retry logging integration
   - Retry with S3Service mock
   - Retry with DocumentGeneratorService integration

## Dev Notes

### Architecture Patterns and Constraints

**Service Pattern:**
- RetryService, generic retry operations sağlar
- Exponential backoff stratejisi ile transient failure'ları handle eder
- Generic type support ile herhangi bir Promise return type'ı destekler
- Configurable retry options ile farklı operasyonlar için farklı retry stratejileri
- [Source: docs/tech-spec-epic-6.5.md#Services-and-Modules]

**Dependency Injection Pattern:**
- NestJS dependency injection kullanılır
- LoggerService required dependency (Epic 7.3'ten)
- RetryService CommonModule üzerinden export edilir
- [Source: docs/tech-spec-epic-6.5.md#System-Architecture-Alignment]

**Exponential Backoff Pattern:**
- Formula: `2^(attempt-1) * baseDelay`
- Delay sequence: 0ms → baseDelay → 2*baseDelay → 4*baseDelay...
- First attempt: Immediate execution (no delay)
- Subsequent attempts: Exponential delay before retry
- [Source: docs/tech-spec-epic-6.5.md#Data-Models-and-Contracts]

**Error Handling Pattern:**
- Retry mechanism handles transient failures automatically
- Final failure throws last error (no silent failures)
- Retry logging helps diagnose persistent failures
- Configurable retry attempts and delays for different operation types
- [Source: docs/tech-spec-epic-6.5.md#Reliability/Availability]

**Module Structure:**
- RetryService: `src/common/services/retry.service.ts`
- RetryOptions interface: `src/common/services/interfaces/retry-options.interface.ts`
- CommonModule export: `src/common/services/index.ts`
- [Source: docs/architecture/project-structure.md#common]

**Import Organization:**
- 8-group import order: Libraries → DTOs → Services → Repositories → Entities → Interfaces → Enums → Events
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Import-Organization]
