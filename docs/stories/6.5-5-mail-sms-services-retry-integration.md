# Story 6.5.5: Mail & SMS Services Retry Integration

Status: done

## Story

As a developer,
I want MailService ve SMSService'te retry mekanizmasını Common RetryService ile entegre etmek,
so that email ve SMS gönderimlerinde geçici network hataları otomatik retry edilsin ve kritik iletişimlerin güvenilirliği artsın.

## Acceptance Criteria

1. **AC-6.5.5.1:** MailService Dependencies Update
   - `src/modules/mail/services/mail.service.ts` güncellendi
   - Constructor'da RetryService inject edilmiş
   - RetryService (Story 6.5-2'den) import edilmiş: `src/common/services/retry.service.ts`

2. **AC-6.5.5.2:** MailService Send Retry Integration
   - sendEmail() method'u RetryService ile wrap edilmiş
   - emailProvider.send() çağrısı retry ile sarmalanmış
   - RetryService.executeWithRetry() kullanılır:
     ```typescript
     const result = await this.retryService.executeWithRetry(
       async () => await this.emailProvider.send(to, subject, html, text),
       {
         maxAttempts: this.mailRetryMaxAttempts, // 5 (configurable)
         baseDelay: this.mailRetryBaseDelay, // 2000ms (configurable)
         context: `Email Send: ${to} - ${subject}`
       }
     );
     ```
   - Max attempts: 5 (critical communication)
   - Base delay: 2000ms (external service)

3. **AC-6.5.5.3:** Template Email Retry Integration
   - sendTemplateEmail() method'u RetryService ile wrap edilmiş
   - Template rendering (templateService.render()) ve email send birlikte retry edilir
   - Retry configuration:
     - maxAttempts: 5
     - baseDelay: 2000ms
     - context: `Template Email: ${templateName} to ${to}`
   - Retry example:
     ```typescript
     await this.retryService.executeWithRetry(
       async () => {
         const html = await this.templateService.render(templateName, data);
         return await this.emailProvider.send(to, subject, html);
       },
       {
         maxAttempts: this.mailRetryMaxAttempts,
         baseDelay: this.mailRetryBaseDelay,
         context: `Template Email: ${templateName} to ${to}`
       }
     );
     ```

4. **AC-6.5.5.4:** SMSService Dependencies Update
   - `src/modules/sms/services/sms.service.ts` güncellendi
   - Constructor'da RetryService inject edilmiş
   - RetryService import edilmiş: `src/common/services/retry.service.ts`

5. **AC-6.5.5.5:** SMSService Send Retry Integration
   - sendSms() method'u RetryService ile wrap edilmiş
   - fonivaService.sendSms() çağrısı retry ile sarmalanmış
   - RetryService.executeWithRetry() kullanılır:
     ```typescript
     const fonivaResult = await this.retryService.executeWithRetry(
       async () => await this.fonivaService.sendSms(phoneNumber, message, type, smsRecord.id),
       {
         maxAttempts: this.smsRetryMaxAttempts, // 5 (configurable)
         baseDelay: this.smsRetryBaseDelay, // 2000ms (configurable)
         context: `SMS Send: ${phoneNumber} - ${message.substring(0, 50)}...`
       }
     );
     ```
   - Max attempts: 5 (critical communication)
   - Base delay: 2000ms (external service)

6. **AC-6.5.5.6:** SMS Retry Method Refactoring
   - Existing retrySms() method REFACTORED to use RetryService
   - Remove manual exponential backoff logic (backoffDelays array)
   - Remove manual setTimeout() delay logic
   - Wrap fonivaService.sendSms() with RetryService:
     ```typescript
     const fonivaResult = await this.retryService.executeWithRetry(
       async () => await this.fonivaService.sendSms(
         smsRecord.phoneNumber,
         smsRecord.message,
         smsRecord.type as SmsType,
         smsRecord.id
       ),
       {
         maxAttempts: 3 - smsRecord.attemptCount, // Remaining attempts
         baseDelay: this.smsRetryBaseDelay,
         context: `SMS Retry: ${smsRecord.id} (attempt ${smsRecord.attemptCount + 1}/3)`
       }
     );
     ```
   - Preserve existing attemptCount tracking logic
   - Keep existing database update logic (status, providerId, errorMessage)

7. **AC-6.5.5.7:** Error Handling - MailService
   - Network timeout errors: Retry (RetryService handles)
   - SendGrid rate limit (429): Retry (RetryService handles)
   - Invalid email (400): Don't retry, throw BadRequestException immediately
   - Retry exhausted: RetryService throws last error, MailService wraps with i18n-based BadRequestException
   - Error classification:
     - Check error code/message before retry
     - Throw BadRequestException for invalid email immediately (skip retry)
     - Let RetryService handle network errors and 5xx errors
   - Preserve existing i18n error messages: `errors.EMAIL_SEND_FAILED`

8. **AC-6.5.5.8:** Error Handling - SMSService
   - Network timeout errors: Retry (RetryService handles)
   - Foniva API errors (5xx): Retry (RetryService handles)
   - Invalid phone number (400): Don't retry, update status to FAILED immediately
   - Retry exhausted: RetryService throws last error, SMSService updates status to FAILED
   - Preserve existing database update logic (status: FAILED, errorMessage, attemptCount)
   - Preserve existing i18n error messages: `errors.SMS_NOT_FOUND`, `errors.SMS_MAX_RETRY_EXCEEDED`

9. **AC-6.5.5.9:** Environment Configuration

## Dev Notes

### Architecture Patterns and Constraints

**Service Integration Pattern:**
- MailService ve SMSService, RetryService'i dependency injection ile alır
- RetryService: Exponential backoff retry (Story 6.5-2'den)
- SMS retrySms() method REFACTORED: Remove manual retry logic (backoffDelays, setTimeout)
- [Source: docs/tech-spec-epic-6.5.md#Integration-Points]

**Retry Pattern:**
- Wrap external service calls (emailProvider.send(), fonivaService.sendSms()) with retry
- Exponential backoff: 2^(attempt-1) * baseDelay
- Max attempts: 5 for email/SMS (configurable)
- Base delay: 2000ms (critical communication, longer than document generation)
- Context-based logging for debugging
- [Source: docs/tech-spec-epic-6.5.md#Retry-Workflow]

**Error Handling Pattern:**
- 4xx client errors (invalid email, invalid phone): Don't retry, throw/update status immediately
- 5xx server errors: Retry via RetryService
- Network timeout errors: Retry via RetryService
- Rate limit errors (429): Retry via RetryService (exponential backoff helps)
- Retry exhausted: MailService throws BadRequestException, SMSService updates status to FAILED
- [Source: docs/tech-spec-epic-6.5.md#Error-Handling]

**Configuration Management:**
- Environment variables: MAIL_RETRY_MAX_ATTEMPTS (default 5), MAIL_RETRY_BASE_DELAY (default 2000ms)
- Environment variables: SMS_RETRY_MAX_ATTEMPTS (default 5), SMS_RETRY_BASE_DELAY (default 2000ms)
- ConfigService ile configuration management
- Higher retry attempts for critical communication (5 vs 3 for document generation)
- [Source: docs/tech-spec-epic-6.5.md#Configuration-Management]

**Module Structure:**
- MailService: `src/modules/mail/services/mail.service.ts`
- SMSService: `src/modules/sms/services/sms.service.ts`
- RetryService: `src/common/services/retry.service.ts` (Story 6.5-2)
- [Source: docs/architecture/project-structure.md#modules]

**Import Organization:**
- 8-group import order: Libraries → DTOs → Services → Repositories → Entities → Interfaces → Enums → Events
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Import-Organization]

### Learnings from Previous Story

**From Story 6.5-4-s3-service-retry-integration (Status: done)**

- **RetryService Usage Pattern:**
  - RetryService.executeWithRetry() ile operation wrap et
  - Retry options: maxAttempts, baseDelay, context
  - Context string for logging: `Email Send: ${to} - ${subject}`
  - Use `RetryService` from `@/common/services/retry.service`
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Dev-Agent-Record]

- **New Service Created - RetryService:**
  - RetryService already implemented in Story 6.5-2
  - Available at: `src/common/services/retry.service.ts`
  - Exported from CommonModule
  - Usage: Inject via constructor: `private readonly retryService: RetryService`
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Learnings-from-Previous-Stories]

- **Exponential Backoff:**
  - Formula: `2^(attempt-1) * baseDelay`
  - Delay sequence: 0ms → 2000ms → 4000ms → 8000ms... (for baseDelay=2000ms)
  - First attempt: Immediate (no delay)
  - Email/SMS operations need longer base delay (2000ms vs 1000ms for document generation)
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Architecture-Patterns]

- **Retry Logging:**
  - Each attempt: `[context] Attempt 1/5`
  - Retry delay: `[context] Waiting 2000ms before retry attempt 2`
  - Success on retry: `[context] Succeeded on attempt 2/5`
  - Final failure: `[context] All 5 attempts failed`
  - Context string critical for debugging
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Logging-Integration]

- **Configuration Pattern:**
  - Environment variables: MAIL_RETRY_MAX_ATTEMPTS, MAIL_RETRY_BASE_DELAY, SMS_RETRY_MAX_ATTEMPTS, SMS_RETRY_BASE_DELAY
  - ConfigService integration with defaults
  - Higher retry attempts for critical communication (5 vs 3)
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Environment-Configuration]

- **Error Handling:**
  - RetryService throws last error after max attempts
  - Wrap RetryService error with custom exception (BadRequestException for MailService)
  - Non-retryable errors (4xx): Throw immediately, skip retry
  - SMSService: Update status to FAILED instead of throwing exception
  - [Source: stories/6.5-4-s3-service-retry-integration.md#Error-Handling-Pattern]

- **Code Quality Improvement:**
  - Remove code duplication (custom retry logic in SMSService.retrySms())
  - Single responsibility principle (RetryService handles retry)
  - Cleaner code, easier to maintain
  - Consistent retry behavior across services

- **I18n Integration:**
  - S3Service integrated I18nService for error messages
  - MailService and SMSService already have I18nService
  - Preserve existing i18n error keys (errors.EMAIL_SEND_FAILED, errors.SMS_NOT_FOUND, etc.)
  - [Source: stories/6.5-4-s3-service-retry-integration.md#File-List]

- **Pending Review Items:** None affecting this story

**Key Takeaways:**
- Story 6.5-5 integrates RetryService into MailService and SMSService
- Refactors SMSService.retrySms() to use RetryService (removes manual backoff logic)
- Higher retry attempts (5) for critical communication vs document generation (3)
- Longer base delay (2000ms) for external service operations
- Error classification: 4xx errors skip retry, 5xx and network errors retry
- Preserve existing error handling patterns (MailService throws, SMSService updates status)
