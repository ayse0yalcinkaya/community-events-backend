# Story 6.5.4: S3 Service Retry Integration

Status: done

## Story

As a developer,
I want S3Service'te retry mekanizmasını Common RetryService ile entegre etmek,
so that file upload/download/delete işlemlerinde geçici network hataları otomatik retry edilsin ve code duplication azalsın.

## Acceptance Criteria

1. **AC-6.5.4.1:** S3Service Dependencies Update
   - `src/modules/files/services/s3.service.ts` güncellendi
   - Constructor'da RetryService inject edilmiş
   - RetryService (Story 6.5-2'den) import edilmiş: `src/common/services/retry.service.ts`
   - Mevcut custom retry logic kaldırılmış (maxRetries, retryDelays, delay() method)

2. **AC-6.5.4.2:** Upload Operation Retry Integration
   - uploadFile() method'u RetryService ile wrap edilmiş
   - RetryService.executeWithRetry() kullanılır:
     ```typescript
     const result = await this.retryService.executeWithRetry(
       async () => await this.s3Client.send(new PutObjectCommand(params)),
       {
         maxAttempts: this.retryMaxAttempts, // 5 (configurable)
         baseDelay: this.retryBaseDelay, // 2000ms (configurable)
         context: `S3 Upload: ${key}`
       }
     );
     ```
   - Max attempts: 5 (network operations benefit from more retries)
   - Base delay: 2000ms (longer delay for external service)

3. **AC-6.5.4.3:** Download Operation Retry Integration
   - getPresignedUrl() method için getSignedUrl() call'ı RetryService ile wrap edilmiş
   - Pre-signed URL generation network işlemi olduğu için retry edilmeli
   - Retry configuration:
     - maxAttempts: 5
     - baseDelay: 2000ms
     - context: `S3 Presigned URL: ${key}`
   - Retry example:
     ```typescript
     const url = await this.retryService.executeWithRetry(
       async () => await getSignedUrl(this.s3Client, command, { expiresIn }),
       {
         maxAttempts: this.retryMaxAttempts,
         baseDelay: this.retryBaseDelay,
         context: `S3 Presigned URL: ${key}`
       }
     );
     ```

4. **AC-6.5.4.4:** Delete Operation Retry Integration
   - deleteFile() method'u RetryService ile wrap edilmiş
   - Retry configuration:
     - maxAttempts: 3 (delete less critical than upload/download)
     - baseDelay: 1000ms
     - context: `S3 Delete: ${key}`
   - Retry example:
     ```typescript
     await this.retryService.executeWithRetry(
       async () => await this.s3Client.send(new DeleteObjectCommand(params)),
       {
         maxAttempts: 3,
         baseDelay: 1000,
         context: `S3 Delete: ${key}`
       }
     );
     ```

5. **AC-6.5.4.5:** Error Handling
   - Network timeout errors: Retry (RetryService handles)
   - Connection errors: Retry (RetryService handles)
   - 5xx server errors: Retry (RetryService handles)
   - 4xx client errors (InvalidBucket, NoSuchKey, AccessDenied): Don't retry, throw immediately
   - Retry exhausted: RetryService throws last error, S3Service wraps with custom exception
   - Error classification:
     - Check error code/message before retry
     - Throw BadRequestException for 4xx errors immediately (skip retry)
     - Let RetryService handle 5xx and network errors

6. **AC-6.5.4.6:** Environment Configuration
   - `.env.example` dosyasına yeni environment variables eklendi:
     ```
     # S3 Service Retry Configuration
     S3_RETRY_MAX_ATTEMPTS=5          # Max retry attempts (network ops)
     S3_RETRY_BASE_DELAY=2000         # Base delay (ms)
     ```
   - ConfigService integration:
     - `S3_RETRY_MAX_ATTEMPTS`: Number, default 5
     - `S3_RETRY_BASE_DELAY`: Number, default 2000

7. **AC-6.5.4.7:** Logging
   - Upload start: `[S3Service] Uploading file: ${key}`
   - Retry attempts: `[S3Service] Retry attempt 2/5 for S3 Upload: ${key}` (RetryService logging)
   - Success: `[S3Service] File uploaded successfully: ${key}`
   - Final failure: `[S3Service] Upload failed after 5 attempts: ${key}`
   - Similar logging for download (presigned URL) and delete operations

8. **AC-6.5.4.8:** testConnection() Method
   - testConnection() (health check) NO RETRY needed
   - Health checks should be fast, no retry overhead
   - Keep existing implementation (single attempt)

9. **AC-6.5.4.9:** Remove Custom Retry Logic
   - Remove private fields:
     - `private readonly maxRetries = 3;`
     - `private readonly retryDelays = [1000, 2000, 4000];`
   - Remove private method:
     - `private delay(ms: number): Promise<void>`
   - Remove manual retry loops in uploadFile() (lines 107-132)
   - Cleaner code, single responsibility principle

10. **AC-6.5.4.10:** Unit Tests
    - Test file: `src/modules/files/services/__tests__/s3.service.spec.ts` güncellendi
    - Test scenarios:
      - Upload successful on first attempt
      - Upload retry on network error (TimeoutError, ECONNRESET)
      - Upload no retry for 4xx errors (InvalidBucket, AccessDenied)
      - Upload retry exhaustion (throw exception after max attempts)
      - Download (presigned URL) retry on network error
      - Delete retry on network error
      - Delete no retry for 4xx errors (NoSuchKey)
    - Mock RetryService

11. **AC-6.5.4.11:** Integration Tests (Optional)
    - Test file: `test/s3.integration-spec.ts` güncellendi/oluşturuldu (varsa)
    - Test scenarios:
      - End-to-end upload with retry on transient failure
      - End-to-end download (presigned URL) with retry
      - End-to-end delete with retry
      - Retry integration with real RetryService

## Dev Notes

### Architecture Patterns and Constraints

**Service Integration Pattern:**
- S3Service, RetryService'i dependency injection ile alır
- RetryService: Exponential backoff retry (Story 6.5-2'den)
- Mevcut custom retry logic kaldırılır (code duplication reduction)
- [Source: docs/tech-spec-epic-6.5.md#Integration-Points]

**Retry Pattern:**
- Wrap external S3 operations (PutObject, GetObject, DeleteObject) with retry
- Exponential backoff: 2^(attempt-1) * baseDelay
- Max attempts: 5 for upload/download (configurable), 3 for delete
- Base delay: 2000ms for upload/download, 1000ms for delete
- Context-based logging for debugging
- [Source: docs/tech-spec-epic-6.5.md#Retry-Workflow]

**Error Handling Pattern:**
- 4xx client errors (InvalidBucket, NoSuchKey, AccessDenied): Don't retry, throw immediately
- 5xx server errors: Retry via RetryService
- Network timeout errors: Retry via RetryService
- Retry exhausted: Throw S3ServiceException with clear message
- [Source: docs/tech-spec-epic-6.5.md#Error-Handling]

**Configuration Management:**
- Environment variables: S3_RETRY_MAX_ATTEMPTS (default 5), S3_RETRY_BASE_DELAY (default 2000ms)
- ConfigService ile configuration management
- Higher retry attempts for network operations (5 vs 3 for document generation)
- [Source: docs/tech-spec-epic-6.5.md#Configuration-Management]

**Module Structure:**
- S3Service: `src/modules/files/services/s3.service.ts`
- RetryService: `src/common/services/retry.service.ts` (Story 6.5-2)
- [Source: docs/architecture/project-structure.md#modules]

**Import Organization:**
- 8-group import order: Libraries → DTOs → Services → Repositories → Entities → Interfaces → Enums → Events
- [Source: docs/architecture/implementation-patterns-ai-agent-consistency-rules.md#Import-Organization]

### Learnings from Previous Story

**From Story 6.5-3-document-generator-cache-retry-integration (Status: done)**

- **RetryService Usage Pattern:**
  - RetryService.executeWithRetry() ile operation wrap et
  - Retry options: maxAttempts, baseDelay, context
  - Context string for logging: `S3 Upload: ${key}`
  - Use `RetryService` from `@/common/services/retry.service`
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Dev-Agent-Record]

- **New Service Created - RetryService:**
  - RetryService already implemented in Story 6.5-2
  - Available at: `src/common/services/retry.service.ts`
  - Exported from CommonModule
  - Usage: Inject via constructor: `private readonly retryService: RetryService`
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Learnings-from-Previous-Stories]

- **Exponential Backoff:**
  - Formula: `2^(attempt-1) * baseDelay`
  - Delay sequence: 0ms → 2000ms → 4000ms → 8000ms... (for baseDelay=2000ms)
  - First attempt: Immediate (no delay)
  - S3 operations need longer base delay (2000ms vs 1000ms for document generation)
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Architecture-Patterns]

- **Retry Logging:**
  - Each attempt: `[context] Attempt 1/5`
  - Retry delay: `[context] Waiting 2000ms before retry attempt 2`
  - Success on retry: `[context] Succeeded on attempt 2/5`
  - Final failure: `[context] All 5 attempts failed`
  - Context string critical for debugging
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Logging-Integration]

- **Configuration Pattern:**
  - Environment variables: S3_RETRY_MAX_ATTEMPTS, S3_RETRY_BASE_DELAY
  - ConfigService integration with defaults
  - Higher retry attempts for network operations (5 vs 3)
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Environment-Configuration]

- **Error Handling:**
  - RetryService throws last error after max attempts
  - Wrap RetryService error with custom exception (ServiceUnavailableException)
  - Non-retryable errors (4xx): Throw immediately, skip retry
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#Error-Handling-Pattern]

- **Code Quality Improvement:**
  - Remove code duplication (custom retry logic)
  - Single responsibility principle (RetryService handles retry)
  - Cleaner code, easier to maintain
  - Consistent retry behavior across services

- **Key Interfaces Created:**
  - ICacheService interface (not needed for S3)
  - IRetryService interface (use concrete RetryService)
  - [Source: stories/6.5-3-document-generator-cache-retry-integration.md#File-List]

- **Pending Review Items:** None affecting this story

**Key Takeaways:**
- Story 6.5-4 replaces S3Service custom retry logic with Story 6.5-2's RetryService
- Reduces code duplication and improves maintainability
- Consistent retry behavior across document generation and S3 operations
- Higher retry attempts (5) for network operations vs document generation (3)
- Longer base delay (2000ms) for external service operations
- Error classification: 4xx errors skip retry, 5xx and network errors retry
